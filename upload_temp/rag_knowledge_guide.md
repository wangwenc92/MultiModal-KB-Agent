# RAG（检索增强生成）知识指南

## 什么是 RAG？

RAG（Retrieval-Augmented Generation，检索增强生成）是一种将信息检索与大型语言模型（LLM）相结合的技术架构。它在 2020 年由 Lewis 等人在论文《Retrieval-Augmented Generation for Knowledge-Intensive NLP Tasks》中首次提出。

## RAG 的核心流程

1. **索引（Indexing）**：将文档切分为文本块（Chunk），通过 Embedding 模型转为向量，存入向量数据库。
2. **检索（Retrieval）**：用户提问时，将问题同样转为向量，在向量库中做相似度搜索（如余弦相似度），找到最相关的 Top-K 个文本块。
3. **生成（Generation）**：将检索到的文本块作为上下文，拼接用户问题，一起送入 LLM 生成回答。

## RAG 的优势

- **解决幻觉问题**：LLM 生成的回答基于检索到的真实文档，大幅降低事实性错误。
- **知识更新灵活**：只需更新向量库中的文档，无需重新训练模型。
- **可追溯来源**：可以展示回答引用的原始文档片段，增强可信度。
- **降低部署成本**：无需针对特定领域微调大模型。

## 常见 RAG 优化技巧

### 分块策略（Chunking）
- **固定大小分块**：按固定 token 数量切分，简单高效。
- **语义分块**：按自然语义单元（段落、句子）切分，保留上下文完整性。
- **递归分块**：先按大粒度切分，再对小块递归细化。

### 检索优化
- **混合检索**：结合向量相似度（语义）和关键词匹配（BM25），提高召回率。
- **重排序（Re-ranking）**：对初筛结果使用 Cross-encoder 重新排序，提升 Top-K 精度。
- **多路召回**：从多个维度（向量、关键词、图结构）分别检索后融合。

### 增强生成
- **提示词压缩**：对检索结果进行摘要，减少 LLM 的上下文长度。
- **自适应检索**：让 LLM 自行判断是否需要检索更多信息。
- **多轮对话中的 RAG**：将历史对话纳入检索上下文。

## RAG vs 微调（Fine-tuning）

| 维度 | RAG | 微调 |
|------|-----|------|
| 知识更新 | 即时更新（换文档即可） | 需要重新训练 |
| 外部知识 | 天然支持 | 难以融入实时信息 |
| 可解释性 | 高（有来源引用） | 低（黑盒） |
| 推理成本 | 较低（仅需检索） | 较高（全参推理） |
| 适用场景 | 知识库问答、客服 | 风格适配、指令遵循 |

## 向量数据库选型

- **ChromaDB**：轻量级，适合原型开发和小规模场景。
- **Pinecone**：云原生，适合生产环境。
- **Weaviate**：支持混合检索和 GraphQL 接口。
- **Milvus / Zilliz**：分布式，适合大规模向量检索。
- **Qdrant**：Rust 编写，性能优秀。

## RAG 评估指标

- **检索质量**：Recall@K、Precision@K、MRR（Mean Reciprocal Rank）
- **生成质量**：忠实度（Faithfulness）、答案相关性（Answer Relevance）、上下文精度（Context Precision）
- **端到端**：整体回答的准确率和用户满意度
